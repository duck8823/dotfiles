# スプリントルール

## スプリント構成
- 基本的にはユーザー価値のあるイシューを必ず1つ以上入れる（該当がない場合は技術的負債解消のみも許容）
- スプリント計画はプロジェクトメモリ（`memory/MEMORY.md`）で管理する。計画用 Issue は作らない
- スプリントの進捗は各タスクの Issue で確認する

## スプリント内の技術的負債
- 実装中に自分で作った技術的負債はそのスプリント内で消化する
- 品質フェーズで発見された技術的負債・不具合は Issue 化し、**同じマイルストーン内の次スプリントに必ず入れる**（現スプリントには入れない、マイルストーンもまたがない）
- tech-debt がゼロになるまでマイルストーン内でスプリントを追加して回す

## スプリント完了時の流れ
1. 全イシューを消化
2. `flutter clean && flutter build ipa` で clean build
3. `~/.zshrc` の認証情報を利用して App Store Connect にアップロード（TestFlight 配信用。リリースではない）
4. マイルストーン最初のスプリント完了時に、リリース用 PR をドラフトで作成（リリースノート更新・バージョン/ビルド番号更新）
5. リリース PR はリリースが完了するまでクローズしない（マイルストーン内の後続スプリントで追記・更新する）
6. 品質フェーズを実施（`~/.claude/skills/quality-phase/SKILL.md` 参照）

## スプリント完了後品質フェーズ

スプリント内の全イシューをマージした後、次のスプリント計画に入る前に以下を実施すること。

### 1. アーキテクチャ・一貫性セルフレビュー（3AI多重）

スプリント全体の変更を俯瞰し、個別 PR レビューでは見えなかった問題を検出する。
Architect ロールを Claude / Codex / Gemini の3AI並列で実行。

```bash
git log --oneline $(git describe --tags --abbrev=0 2>/dev/null || git rev-list --max-parents=0 HEAD)..HEAD
git diff $(git describe --tags --abbrev=0 2>/dev/null || git rev-list --max-parents=0 HEAD)..HEAD --stat
```

### 2. 探索的テスト（自律 QA）

アプリを実際に起動し、Claude Code が QA エンジニアとして自律的にシナリオテストを行う。

**テスト対象シナリオの導出:**
スプリントでクローズした Issue の受け入れ条件および関連 PR の説明からテストシナリオを列挙する。既存機能のリグレッションシナリオも含める。

**チェック観点:**
- スプリントで実装した機能の受け入れ条件を全て満たしているか
- 正常系・異常系・境界値・空状態・ローディング状態
- 権限・認証まわりのセキュリティ抜けがないか
- データの整合性（削除連鎖・バリデーション等）

### 3. デザインチェック

プロジェクトの CLAUDE.md にデザインシステム・Figma・Storybook 等の参照先が記載されていれば参照して確認する。記載がない場合は「デザインシステム参照不可」と明記したうえで、コードから読み取れる範囲のみチェックする。推測でデザイン適合を判断しない。

### 4. ガベージコレクション（整合性チェック）

コードベースとハーネス設定の不整合を定期検出する。

- 未使用の import・dead code の検出（`flutter analyze` / linter で検出可能な範囲）
- プロジェクト CLAUDE.md のパス・コマンドが実際のプロジェクト構造と一致しているか確認
- メモリファイル（`memory/`）の鮮度チェック: 30日以上更新なし → 内容を検証し、陳腐化していれば更新または削除

### 5. 問題のトリアージと対応

| 重大度 | 基準 | 対応 |
|--------|------|------|
| **クリティカル** | バグ・セキュリティ脆弱性・データ破損リスク・クラッシュ | 即座に修正して当スプリント内でマージ |
| **その他** | 技術的負債・改善・一貫性・デザイン調整・テスト追加 | `gh issue create` で登録し**同マイルストーン内の次スプリントに必ず入れる** |
