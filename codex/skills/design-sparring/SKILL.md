# Design Sparring

## 入力
- 背景
- 目的
- 制約（納期・性能・運用・互換性）
- 候補案（A/B 以上）
- 評価軸（保守性・速度・コストなど）

不足があれば先に埋める。

## 手順
1. 前提・非目標を固定する。
2. 候補案を同じ粒度で比較する。
3. 各案の利点・欠点・リスク・回避策を列挙する。
4. 評価軸ごとに判断根拠を示す。
5. Structure-Behavior の観点で、概念・責務・境界・振る舞いテストへの影響を確認する。
6. 推奨案を1つ選び、採用条件と却下条件を明示する。

## Structure-Behavior チェック
- 中心概念・状態・不変条件は何か
- どの責務がどの owner に属するか
- handler / usecase / service は orchestration に留まるか
- どの詳細を boundary / interface で隠すか
- 観測可能な振る舞い・状態遷移・エラーをどのテストで守るか
- premature abstraction と抽象化不足のどちらが主リスクか

## 出力フォーマット
- 結論（推奨案）
- 理由（評価軸ごとの比較）
- 概念・責務・境界・テスト観点の判断
- リスクと対策
- 採用条件 / 却下条件
- 次のアクション（実装前チェック項目）

## 禁止
- 根拠なしの断定
- 比較を省略して好みで決める
- いきなり実装に飛ぶ
